Quel est le but de WAI-ARIA et comment fonctionne-t-il ?

Rendre accessible du contenu statique peut être relativement simple, mais rendre accessible du contenu dynamique peut s'avérer plus complexe. C'est là qu'intervient WAI-ARIA.

WAI-ARIA signifie "Web Accessibility Initiative - Accessible Rich Internet Applications"

Il s'agit d'une spécification qui améliore l'accessibilité des contenus dynamiques et des composants d'interface utilisateur

WAI-ARIA fonctionne en introduisant un ensemble d'attributs que vous pouvez ajouter aux éléments HTML pour fournir des informations sémantiques supplémentaires. Ces attributs sont classés en rôles, états et propriétés.

Le rôle ARIA définit la fonction d'un élément au sein d'un site web ou d'une application web

        
            <div role="button">Click me</div>

Ce faisant, vous indiquez aux technologies d'assistance que l'élément est un bouton. Cependant, les roles n'offrent aucune fonctionnalité, le simple fait de lui attribuer div un rôle rôle button ne le fera pas agir comme un button, pour qu'il le fasse il faut integrer du CSS et JS apres il est préferable d'utiliser l'élément natif. button ou Input l'élement avec type="button".

L'état ARIA décrit l'état actuel d'un élément, qui peut évoluer en fonction des interactions de l'utilisateur. L' aria-expanded attribut des sections réductibles en est un exemple

Encore une fois, il s'agit simplement d'une description de l'état ; aucune fonctionnalité n'est fournie. Vous devrez utiliser JavaScript pour modifier l'état de l'attribut lorsque l'utilisateur clique sur le bouton.

Les propriétés ARIA fournissent des détails supplémentaires sur les éléments. Par exemple, elles aria-labelledby permettent de lier un élément à une étiquette spécifique :

Pour tirer le meilleur parti de WAI-ARIA, essayez de vous en tenir au HTML natif autant que possible, car il offre généralement plus d'accessibilité dès le départ.

Utilisez WAI-ARIA uniquement lorsque le HTML est insuffisant, et n'oubliez pas de tester votre travail avec des technologies d'assistance comme des lecteurs d'écran, ou de faire tester votre travail par des personnes handicapées.

Quel sont les rôles ARIA

ARIA signifie Accessible Rich Internet Applications.

Les rôles ARIA précisent la signifiation sémantique des éléments HTML. Ils sont essentielle pour les personnes utilisant des technologies d'assistance, comme les lecteurs d'écran.

les sémantique ont déja un rôle ARIA attribué par défault alors que les élement non sémantique tels que <div> n'ont pas de rôle çela signifie que les lecteurs d'écran ne sauront pas l'interpreter l'objectif d'un élément <div> si on n'explique pas explicitement sont rôle.

Pour spécifier le rôle ARIA d'un élement. Il nous suffit d'ajouter role attribut comme çeci role="ARIA role" ou value est le nom d'un rôle dans la spécification ARIA.

à noter que la spécification d'un rôle sur un élément n'a qu'un seul effet: elle informe la technologie d'assistance de la fonction de l'élément.

c'est au dévelopeur d'ajouter un comportement au rôle

Voici les six catégories principales de rôles Aria

  • Rôles de la structure du document
  • Rôles des widgets
  • Rôles marquants
  • Rôles de la région en direct
  • Rôles de fenêtre
  • les rôles abstraits

Les rôles de structure : de document définissent la structure globale de la page web. Grâce à ces rôles, les technologies d'assistance peuvent comprendre les relations entre les différentes sections et aider les utilisateurs à naviguer dans le contenu.

Cependant, la plupart des rôles de structure de document ne sont pas utilisés dans le développement Web moderne, car les navigateurs prennent déjà en charge des éléments HTML sémantiques équivalents, qui doivent être prioritaires dans la mesure du possible.

Les rôles des widgets : définissent l'objectif et la fonctionnalité des éléments interactifs, comme les barres de défilement.

Les rôles de repère(marquants) : classent et étiquettent les sections principales d'une page web. Les lecteurs d'écran les utilisent pour faciliter la navigation vers les sections importantes d'une page. Utilisez-les avec parcimonie afin de conserver une présentation générale simple et compréhensible

Les rôles de région active : définissent des éléments dont le contenu évolue de manière dynamique. Ainsi, les lecteurs d'écran et autres technologies d'assistance peuvent annoncer les modifications aux utilisateurs malvoyants

Les rôles de fenêtre : définissent des sous-fenêtres, comme des boîtes de dialogue modales contextuelles

les rôles abstraits : Ces rôles aident à organiser le document. Ils sont uniquement destinés à être utilisés en interne par le navigateur, et non par les développeurs. Vous devez donc savoir qu'ils existent, mais vous ne devez pas les utiliser sur vos sites web ou applications web.

Quels sont les rôles des attributs aria-labelet ?aria-labelledby

les attributs aria-label et aria-labelledby fournissent des informations cruciales sur les éléments de la page qui peuvent être flous ou invisibles.

aria-label est particulièrement utile pour les éléments sans texte visible, mais nécessitant néanmoins une description par les lecteurs d'écran. Par exemple, les boutons composés uniquement d'icônes doivent souvent aria-label communiquer leur fonction.

aria-labelledby a la même fonction que aria-label attribut, mais au lieu de définir le texte directement dans l'attribut, on utilise une référence à un texte existant sur la page Le texte existant doit posséder un id attribut, qui servira de valeur de référence dans l' aria-labelledby attribut

example :

            
                <input type="text" aria-labelledby="search-btn">           
                <button type="button" id="search-btn">Search</button>

le texte du bouton sert d'étiquette pour la recherche. Les lecteurs d'écran annonceront l'entrée sous la forme Search, edit. Si vous décidez ultérieurement de remplacer le texte du bouton par Find, l'étiquette de l'entrée sera automatiquement mise à jour avec le nouveau texte, car elle fait référence au texte du bouton

  • Si quelqu'un utilise un service de traduction pour traduire le contenu de votre page, le texte d'un aria-label attribut peut ne pas toujours être traduit.
  • L'utilisation aria-labelledby peut également aider à éviter une incompatibilité entre le texte de l'étiquette visible et l'étiquette invisible pour les utilisateurs de lecteurs d'écran, car la mise à jour du texte visible mettra automatiquement à jour l'étiquette invisible.
  • aria-labelledby peut rendre beaucoup plus facile la création par programmation d'étiquettes invisibles complexes constituées de plusieurs sources de texte.
  • Tips : n'utilisez pas aria-label « and » aria-labelledby simultanément sur un même élément. Dans ce cas, l'étiquette invisible pour les lecteurs d'écran sera toujours déterminée par aria-labelledby et aria-label sera totalement ignorée.

Qu'est-ce que l'attribut aria-hidden et comment fonctionne-t-il ?

Si on a besoin d'afficher du contenu tout en le masquant aux personne qui utilisent des technologie d'assistance (lecteurs d'ecran), on peut utiliser l'aria-hidden

utilisation : il nous suffit d'ajouter à l'élément HTML qu'on souhaite masquer et de définir ça valeur sur TRUE comme on peut le voir la : aria-hidden="true"

Cet attribut masque l'élément et tous ses enfants de l'arborescence d'accessibilité, tout en les maintenant visibles sur la page

on peut les utiliser pour

  • Icônes et images qui ont uniquement un but décoratif.
  • Contenu dupliqué.

si le contenu doit être masquer à tous et pas seulement aux techno d'assitance on utiliser display CSS à none sur le menu pour le supprimer lorsqu'il est réduit.

ne jamais utiliser aria-hidden pour masquer un élément Accesible aux clavier car il ne supprime pas du DOM. ainsi les utilisateurs pouront acceder à l'element via la touche Tab mais leur lecteur d'ecran ne l'annoncera pas.

Example : où nous masquons une icône de l'arborescence d'accessibilité en ajoutant l' aria-hiddenattribut avec une valeur de true.

Nous gardons uniquement le texte exposé aux technologies d'assistance pour éviter toute confusion qui pourrait résulter de la redondance d'avoir à la fois une icône et un texte pour le même objectif.

            
                <button>
                    <i class="fa-solid fa-gear" aria-hidden="true"></i>
                    <span class="label">Settings</span>
                </button>

Vous n'avez pas besoin de l'utiliser aria-hidden lorsque :

  • L'élément HTML possède déjà un hidden attribut.
  • L'élément ou l'ancêtre de l'élément est déjà masqué avec display: none.
  • L'élément ou l'ancêtre de l'élément est déjà masqué avec visibility: hidden.

Dans ces trois cas, l'élément est déjà supprimé du DOM et donc masqué de l'arbre d'accessibilité, l'aria-hidden attribut n'est donc pas nécessaire.

Comme pour l'utilisation de tous les attributs ARIA, vous devez toujours tester avec des technologies d'assistance, et de préférence en demandant à des personnes handicapées de tester votre travail,

Vous devez également savoir que le paramètre aria-hidden sur false n'exposera pas l'élément aux technologies d'assistance si l'un de ses parents a cet attribut défini sur true.

Son utilisation aria-hidden est très limitée et doit principalement se limiter à améliorer l'expérience des utilisateurs de lecteurs d'écran en supprimant les informations purement décoratives ou dupliquées.

Quel est l’ aria-expanded attribut et comment fonctionne-t-il ?

cet attribut est utilisé à des fins d'accessibilité pour indiquer si un contrôle est déplié ou replié. Il est utilisé conjointement avec des widgets repliables tels que les menus, les accordéons et autres widgets de divulgation qui contrôlent la visibilité du contenu

cet attribut est définit sur true si le controle est déplié ou false replié

Quel est l'aria-live attribut et comment fonctionne-t-il

crée une région en direct sur votre page qui peut être utilisée pour informer les utilisateurs de lecteurs d'écran des modifications de contenu dynamiques dans la région en direct au fur et à mesure qu'elles se produisent

cette valeur, assertive cela signifie que la mise à jour est très importante. Elle a la priorité la plus élevée ; l'utilisateur doit donc être averti immédiatement. cette assertivevaleur ne doit donc être utilisée que pour les notifications urgentes ou critiques.

La valeur suivante dans l’ordre de priorité est polite. Cette valeur signifie que la mise à jour n'est pas urgente ; le lecteur d'écran peut donc attendre la fin de l'annonce en cours ou la fin de la saisie de l'utilisateur avant d'annoncer la mise à jour. La plupart des régions actives utiliseront cette polite valeur.

La valeur de priorité la plus basse est off ce qui signifie que la mise à jour ne sera annoncée que si le contenu se trouve dans un élément actuellement activé par le clavier. En réalité, cette valeur n'est presque jamais utilisée, car son utilisation est très limitée et elle n'est pas implémentée de manière cohérente (voire inexistante) sur tous les lecteurs d'écran. Si vous avez besoin de régions dynamiques, prévoyez de l'utiliser polite pour tout, sauf pour les messages critiques qui nécessitent assertive.

Quels sont les états ARIA courants utilisés sur les éléments de contrôle personnalisés ?

es éléments de contrôle de forme sémantique tels que input, select, textarea, button, et fieldset ont des états intégrés qui sont transmis aux technologies d'assistance.

aria-selected Cet état indique qu'un élément est sélectionné. Vous pouvez l'utiliser sur des contrôles personnalisés comme une interface à onglets, une liste déroulante ou une grille.L' aria-selected état indique l'onglet actuellement sélectionné.

aria-disabled Cet état sert à indiquer qu'un élément est désactivé uniquement pour les personnes utilisant des technologies d'assistance, comme les lecteurs d'écran

aria-haspopup Cet état indique qu'un élément interactif déclenchera une fenêtre contextuelle lorsqu'il sera activé.

Vous ne pouvez utiliser cet aria-haspopup attribut que si la fenêtre contextuelle possède l'un des rôles suivants : menu, listbox, tree, gridou dialog

aria-required permet d'indiquer qu'un champ doit être renseigné avant la soumission du formulaire

Nous devons utiliser l'contenteditable attribut pour permettre aux utilisateurs de saisir leurs données.

aria-checked indique si un élément est coché

Quel est l’ aria-controls attribut et comment fonctionne-t-il ?

permet de créer une relation programmatique entre un élément contrôlant la présence d'un autre élément sur la page et l'élément qu'il contrôle. Cette relation peut aider les utilisateurs de lecteurs d'écran à mieux comprendre le fonctionnement du contrôle

Parmi les utilisations courantes, on peut citer un ensemble d'onglets contrôlant la visibilité de différentes sections de contenu, ou un bouton permettant d'activer ou de désactiver la visibilité d'un menu

Quel est l’ aria-describedby attribut et comment fonctionne-t-il ?

permet de fournir des informations complémentaires sur un élément aux utilisateurs de lecteurs d'écran en référençant le contenu existant sur la page. Il crée une association programmatique entre l'élément et le contenu (techniquement appelée description accessible), que les lecteurs d'écran peuvent utiliser pour informer les utilisateurs de ces informations complémentaires lorsqu'ils interagissent avec l'élément.

l'utilisation la plus courante est d'associer des instructions et des messages d'erreur aux champs de formulaire. permet de garantir qu'ils les entendront.

autre cas d'utilisation est la suppression d'un élément button suivi d'un message décrivant ce qui se passe après un click

Quand l’alt attribut est-il nécessaire

est un texte alternatif souvent abrégée en alt, est une brève description d'une image. Il fournit des informations essentielle sur l'image aux utilisateurs qui ne peuvent pas la voir.

Un bon exemple de texte alternatif serait : « Un chiot noir et blanc avec un collier orange est allongé sur le ventre dans le sable, regardant de côté. Une balle orange vif repose près de ses pattes avant. »

Les meilleurs pratique pour rédiger un texte pour l'attribut alt :

  • Essayez de rédiger alt un texte court. Il doit être suffisamment détaillé pour permettre de comprendre l'image, mais pas trop long pour ne pas prêter à confusion.
  • N'essayez pas de décrire chaque détail. Concentrez-vous sur les aspects les plus importants de l'image.
  • En général, il n'est pas nécessaire de commencer par « image de » ou « photo de ». Vous pouvez simplement commencer la description directement.
  • De plus, s'il y a un texte similaire autour de l'image, vous n'avez pas besoin de l'écrire à nouveau.
  • Il est généralement recommandé de terminer le texte alternatif par un point pour plus de cohérence.
  • Si l'image est un lien vers une autre page, au lieu de décrire l'image elle-même, le alttexte doit décrire ce qui se passera si les utilisateurs cliquent dessus.

Seule les images transmettant des informations importantes doivent contenir un alt les images servant uniquement de décoration n'en n'ont pas besoin pour éviter que les lecteurs d'ecran s'attarde sur des informations non essentielle

Toute vos images doivent contenir un alt car si on l'omet les lecteurs d'écran pourront lire le nom du fichier c'est pour ça qu'on met alt en null

Enfin, avant la mise en ligne de votre site Web, vous devez soigneusement tester si les lecteurs d’écran peuvent lire altcorrectement le texte.

Quels sont les avantages d’un bon texte de lien en termes d’accessibilité

il permet à chacun de trouver rapidement l'information. Des liens descriptifs aident les utilisateurs à savoir où ils vont et à quoi ils vont accéder. Cela évite que l'utilisateur se sente perdu et améliore l'expérience utilisateur globale

Pour ceux qui utilisent des lecteurs d'écran, un texte de lien clair et descriptif est indispensable. Les lecteurs d'écran lisent le texte du lien à voix haute ; un texte comme « Lire notre guide d'accessibilité » est donc bien plus judicieux que « Cliquez ici ».

voici quelque bonne pratique lors de la rédaction de texte de lien.

  • Assurez-vous que les liens sont visuellement distincts en utilisant le soulignement et d’autres repères visuels, afin que les utilisateurs puissent facilement les identifier et les parcourir.
  • Évitez les textes de liens génériques tels que « ici », « cliquez ici » et « plus d'informations », car ils ne fournissent aucune information utile.
  • Visez des textes de liens concis et descriptifs, idéalement entre 2 et 5 mots, qui transmettent l'objectif du lien.
  • Évitez le jargon et les abréviations que les utilisateurs pourraient ne pas comprendre.
  • Concentrez-vous sur la destination, et non sur l'action. Par exemple, « Résultats du comportement utilisateur » plutôt que « Cliquez ici pour en savoir plus ».
  • Ne répétez pas le même texte de lien pour différentes destinations.
  • Placez les liens de manière à ce qu'ils soient cohérents avec le texte environnant. Par exemple, « Pour plus d'informations, visitez notre page d'événements » au lieu de « Cliquez ici pour en savoir plus ».

Une vidéo ne se résume pas seulement à des éléments visuels, mais aussi à de l'audio. La première chose à laquelle vous devriez songer est donc d'ajouter des légendes ou des sous-titres à votre contenu vidéo.

Utilisez track element à l'intérieur de notre element video ou audio pour ajouter des légendes ou des sous-titre

kind attribut permet d'indiquer à l'élément de piste comment il doit être utilisé. ces valeur sont : caption, subtitles, chapters, metadata

srclang attribut represente la langue du track contenu

Il est également important de fournir une transcription de vos contenus audio et vidéo. Une transcription est une version textuelle de tous les mots prononcés dans votre audio ou vidéo. Contrairement aux sous-titres, les transcriptions n'ont pas besoin d'être synchronisées avec le média.

D'autre moyen pour rendre votre contenu accessible est d'utiliser :

  • Ajout d'une superposition en langue des signes pour les vidéos destinées aux personnes sourdes et malentendantes.
  • Fournit des contrôles de volume et de vitesse.
  • Assurer un bon contraste pour le texte à l’écran.

Quelles sont les différentes manières de rendre les applications Web accessibles au clavier ?

Cet tabindex attribut permet de rendre les éléments accessibles et de définir l'ordre de navigation au clavier. Il prend une valeur numérique, positive, nulle ou négative.

Il est important de ne jamais utiliser tabindex attribut avec une valeur supérieure à 0. Il est préférable d'utiliser une valeur égale 0 ou supérieure à -1

accesskey est un autre attribut permettant de rendre le clavier de votre projet web accessible. Il permet de définir une touche qui cible ou active un élément particulier :

        
            <button accesskey="s">Save</button>
            <button accesskey="c">Cancel</button>
            <a href="index.html" accesskey="h">Home</a>

Dans le code ci-dessus :

  • accesskey="s" Affecte la touche S au Save bouton. Sur la plupart des navigateurs, appuyer sur ALT + S(sous Windows) et CTRL + Option + S(sous Mac) active ce bouton.
  • accesskey="c" définit la clé C du Cancel bouton, permettant aux utilisateurs de l'activer à l'aide de ALT + C(Windows) et CTRL + Option + C(Mac).
  • accesskey="h" attribue la clé H au Home lien, permettant aux utilisateurs de naviguer vers la page d'accueil en utilisant ALT + H(Windows) et CTRL + Option + H(Mac).

Une autre façon de rendre le clavier accessible dans vos applications est de fournir des indicateurs de focus clairs. Si l'indicateur de focus par défaut du navigateur vous semble insuffisant, vous pouvez le remplacer en ciblant l'état de focus de l'élément.

Example

        
            element:focus {
                outline: 2px solid #005fcc;
                          }

Cette outline propriétés permet de définir le contour de l'élément. cet example définit le contour comme une ligne bleu

L'indicateur de focus doit être stylé de manière à indiquer clairement l'élément actuellement actif. Pour être accessible, l'indicateur doit présenter un contraste de couleur d'au moins 3:1 avec la couleur d'arrière-plan qu'il recouvre.

Vous devez également éviter les pièges du clavier, qui se produisent lorsqu'un utilisateur ne peut pas déplacer le focus d'un certain élément dans des composants tels que les modaux et les fenêtres contextuelles.
